# Project Context

- **Owner:** Ricky Kaare Engelharth
- **Project:** Cabazure.Client — a .NET library for generating strongly-typed HTTP clients via C# Source Generators
- **Stack:** C# / .NET 9, netstandard2.0 targets, IIncrementalGenerator, Roslyn (Microsoft.CodeAnalysis.CSharp 4.14), xunit v3, Verify snapshot testing, Azure.Core / Azure.Identity
- **Created:** 2026-03-10

## Learnings

- Two source projects: `src/Cabazure.Client/` (generator) and `src/Cabazure.Client.Runtime/` (runtime)
- Generator entry points: `ClientEndpointGenerator` and `ClientInitializationGenerator` — both implement `IIncrementalGenerator`
- Descriptor classes in `src/Cabazure.Client/Descriptors/` handle all AST analysis
- `TypeConstants.cs` is the single source of truth for fully-qualified attribute type names
- `IsExternalInit.cs` and `NullableAttributes.cs` polyfill C#9+ features for netstandard2.0
- Validation errors are emitted as Roslyn diagnostics via `context.ReportDiagnostic()` — never throw exceptions
- Generated files must start with `// <auto-generated/>` and include `#nullable enable`
- Build: `dotnet build -c Release` | Test: `dotnet test -c Release`
- `global.json` SDK version must align with GitHub Actions workflows (`.github/workflows/*.yml`). CI uses `dotnet-version: '10.0.x'`, so global.json should pin to `10.0.0` with `rollForward: latestMinor`. Local SDK mismatches are acceptable for dev machines, but CI must match.
- The project targets .NET 10 for test/sample projects, but runtime libraries target netstandard2.0. The generator targets netstandard2.0 (Roslyn SourceGenerator constraint).
- `LangVersion=Latest` is hoisted to `Directory.Build.props` and applies solution-wide. Do not override in individual project files unless there's a specific reason (e.g., AzureRest sample demonstrating older C# version compatibility).
- TFM upgrades should be atomic: global.json + CI workflows + test projects + sample projects, committed together with conventional commit message.
